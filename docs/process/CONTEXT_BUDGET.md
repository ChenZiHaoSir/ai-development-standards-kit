# Context Budget

本规范吸收 `rtk` 的核心思想：AI 开发不是把所有命令输出和文件内容全部塞进上下文，而是优先给模型“足够决策的信息”。目标是节省 token、降低噪声、减少误判，并保留必要的可追溯证据。

## 基本原则

- 先摘要，再展开。
- 先失败信息，再完整日志。
- 先文件结构，再读具体文件。
- 先搜索定位，再读取相关片段。
- 先 diff 摘要，再读取完整 diff。
- 长输出必须截断、分组、去重或保存到文件后引用路径。
- 不把密钥、完整用户数据、完整 prompt、完整日志正文塞进上下文。

## 推荐工具

`rtk` 是推荐的可选增强工具，不是项目运行时依赖。它会压缩命令输出，减少进入 AI 上下文的噪声；不会改变源码、构建产物或业务逻辑。

如果环境没有安装 `rtk`，AI 可以在用户授权或项目规范允许的情况下自动安装：

```bash
./scripts/setup-rtk.sh
```

安装策略：

- macOS / Linux 且存在 Homebrew：使用 `brew install rtk`。
- 没有 Homebrew：输出官方安装命令，由用户或执行环境决定是否运行。
- 安装失败时不得阻塞开发，必须降级使用普通命令和截断规则。
- 不把 `rtk` 写入项目业务依赖、生产依赖或锁文件，除非项目本身就是工具链项目且明确需要。

如果环境已安装 `rtk`，优先使用 token 优化命令：

```bash
rtk git status
rtk git diff
rtk git log -n 10
rtk read path/to/file
rtk grep \"pattern\" .
rtk find \"*.ts\" .
rtk pytest
rtk pnpm test
rtk tsc
rtk lint
rtk gain
```

可在 Codex 环境初始化 RTK 辅助配置：

```bash
rtk init --codex --dry-run
rtk init --codex
```

如果当前 AI 工具支持全局 hook，并且用户允许修改全局配置，可以使用：

```bash
rtk init --global
```

如果没有 `rtk`，使用普通命令时必须控制输出：

```bash
rg --files
rg -n \"pattern\" .
sed -n '1,220p' file
git diff --stat
git diff --name-only
git log --oneline -n 20
pnpm test -- --runInBand 2>&1 | tail -120
```

## 命令输出规则

| 场景 | 优先输出 |
| --- | --- |
| 目录探索 | 文件列表、目录摘要，不直接全量 tree |
| 搜索 | 文件名、行号、短上下文 |
| 读文件 | 相关片段，必要时分段读取 |
| Git 状态 | 变更文件和状态 |
| Git diff | 先 `--stat` / `--name-only`，再读关键文件 diff |
| 测试 | 失败用例、错误堆栈、摘要 |
| 构建 | 错误和警告，隐藏成功噪声 |
| 日志 | 最近错误、去重后的错误类型、时间范围 |
| 依赖 | 新增/删除/高风险依赖摘要 |

## 长输出处理

当命令可能输出大量内容时：

1. 先运行摘要命令。
2. 如果需要完整输出，保存到临时或报告文件。
3. 在主会话只引用文件路径、摘要、关键行和未覆盖风险。
4. 不把完整日志直接粘进对话。

示例：

```bash
pnpm test > reports/test-output.txt 2>&1
tail -120 reports/test-output.txt
```

## 多智能体上下文规则

主会话给子智能体分配任务时，只传必要上下文：

- `/goal`
- 相关文件列表
- 关键片段或接口契约
- 验收标准
- 允许修改和禁止修改的文件范围
- 已知风险

不要把整个 PRD、整个仓库结构、完整日志、全部 diff 同时丢给每个子智能体。子智能体需要更多上下文时，应明确请求具体文件或范围。

## 证据保留

节省 token 不能牺牲可追溯性。以下内容必须保留：

- 失败测试的关键错误。
- 安全扫描的关键告警。
- 发布验证的命令和结论。
- 性能对比的指标。
- 原始长日志的保存路径。

使用 `rtk` 时仍然必须遵守：

- 涉及数据迁移、权限、安全、计费、发布、删除、回滚等高风险判断时，必须读取关键原文或完整 diff 片段验证。
- 测试失败、构建失败、安全扫描失败时，可以先看 `rtk` 摘要，但最终报告必须包含失败命令、关键错误和原始日志路径。
- 如果 `rtk` 摘要与原始输出存在冲突，以原始输出为准。

## 禁止事项

- 禁止无目的读取整个仓库。
- 禁止把大型构建日志、测试日志、锁文件、压缩产物完整塞进上下文。
- 禁止为了省 token 删除安全、测试、发布和审计证据。
- 禁止只看摘要就做高风险结论；涉及数据丢失、安全、迁移、发布时必须抽样或读取关键原文验证。
