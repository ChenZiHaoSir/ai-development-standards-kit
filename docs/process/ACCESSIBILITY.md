# 可访问性 (Accessibility)

本文件定义产品必须达到的可访问性标准，采用 WCAG 2.1 AA 级别。

## 核心原则

- 可访问性是设计和开发的基本要求，不是上线后的补丁。
- 所有新功能必须满足可访问性基础要求，不依赖后续"有时间再补"。
- 颜色不能单独表示状态，必须配合图标、文字或图案。

## HTML 基础

- 每个页面必须有 `<title>` 标签，准确描述页面内容。
- 所有页面必须有层级清晰的 `<h1>` → `<h6>`，不允许跳级。
- 所有图片必须有 `alt` 属性；装饰性图片使用 `alt=""`。
- 所有表单字段必须有 `<label>` 或 `aria-label`。
- 所有交互元素必须是可聚焦的（tabindex 合理设置）。
- 所有交互元素必须支持键盘操作（Enter、Space、Escape、Arrow）。

## 颜色与对比

- 文本颜色和背景色对比度：普通文本 ≥ 4.5:1，大号文本 ≥ 3:1。
- 非文本元素对比度 ≥ 3:1。
- 状态不能只用颜色表示：必须有图标、文字或边框辅助。
- 颜色盲模式：链接不能只用颜色识别，必须有下划线或其他标识。

## 键盘导航

- 所有交互功能必须可键盘操作。
- 焦点顺序必须符合视觉阅读顺序（从左到右，从上到下）。
- 弹窗必须支持 Escape 关闭，焦点必须捕获在弹窗内。
- 菜单、下拉、标签页必须支持箭头键导航。
- 跳过导航区的链接：提供"跳到主内容"的快捷方式。

## 表单

- 错误信息必须清晰描述问题（如"邮箱格式不正确"而不是"字段错误"）。
- 错误信息必须关联到对应字段（使用 `aria-describedby` 或 `aria-errormessage`）。
- 必填字段必须有视觉标识和 `aria-required="true"`。
- 表单提交后必须有明确的成功/失败反馈。

## 图像与媒体

- 动画必须可暂停（提供暂停按钮或 `prefers-reduced-motion` 支持）。
- 视频必须有字幕或文字稿。
- 图片必须有有意义的 alt 文本，不能使用"image"、"img"等泛化描述。
- SVG 必须有 `role="img"` 和 `aria-label`。

## 文本可访问性

- 禁止使用纯图片文本（如截图中的文字）。
- 文本必须可放大到 200% 而不丢失信息。
- 行间距、段落间距必须可通过浏览器设置调整。
- 不使用 `user-select: none` 限制文本选择。

## 触屏/移动端

- 可点击区域最小 44x44px（iOS）/ 48x48dp（Android）。
- 长按必须有备选交互方式。
- 手势操作（滑动、捏合）必须有非手势备选。

## 无障碍标签

- 所有交互元素必须有无障碍标签。
- 图标按钮必须有 `aria-label`。
- 角色切换的组件必须有 `role` 属性。
- 动态更新内容必须使用 `aria-live` 区域。

## 测试要求

- 必须用屏幕阅读器（如 VoiceOver、NVDA）测试核心路径。
- 必须用键盘完整操作核心路径。
- 必须用 Lighthouse Accessibility 审计，评分 ≥ 90。
- 必须在 200% 缩放下测试文字是否截断或重叠。

## 禁止行为

- 禁止用 `div` 或 `span` 替代语义化标签（button、input、a、nav、main、article）
- 禁止颜色单独表示状态（错误不能只有红色边框，必须有图标或文字说明）
- 禁止图片 alt 为空（除装饰性图片外）
- 禁止表单错误只在 console.log，不在界面显示
- 禁止使用 `display: none` 隐藏可访问内容
- 禁止在移动端只支持触摸不支持键盘操作
