# Performance Baseline

本文件定义项目性能底线。即使项目不大，也必须避免明显卡顿、慢响应、重复请求、阻塞渲染和无边界数据加载。

## 核心原则

- 性能不是上线后的补丁，必须在技术选型、架构、接口、开发和 QA 阶段持续检查。
- 项目规模小不代表可以忽略性能；小项目也不能出现明显卡顿、页面冻结、重复加载或接口无响应。
- 全栈一体不是默认安全选项。选择全栈一体时，必须说明服务端渲染、API、数据库、缓存、会话、构建产物和部署环境对性能的影响。
- 所有核心用户路径必须有可观测的耗时、错误和加载状态。
- 性能优化优先处理真实瓶颈，不为了猜测提前引入复杂缓存、队列、微服务或过度拆分。

## 性能预算

项目启动阶段必须根据产品类型设定性能预算。默认预算如下，可按项目调整：

| 指标 | 默认目标 | 阻塞线 |
| --- | --- | --- |
| 首屏可交互 | 2.5 秒内 | 超过 4 秒必须阻塞发布 |
| 路由切换反馈 | 200 毫秒内出现加载/骨架/状态反馈 | 超过 1 秒无反馈必须修复 |
| 常规 API 响应 | P95 小于 500 毫秒 | P95 超过 1.5 秒必须定位 |
| 列表查询 | P95 小于 800 毫秒 | 无分页/无限制查询阻塞发布 |
| 用户操作反馈 | 100 毫秒内有视觉反馈 | 点击后无反馈阻塞核心流程 |
| 构建产物体积 | 按框架建议预算控制 | 明显超预算必须解释和拆分 |
| 主线程长任务 | 避免超过 50 毫秒 | 频繁长任务必须优化 |

## 全栈一体性能风险

选择 Next.js、Nuxt、Laravel、Rails、Django、Blitz、Remix 等全栈一体方案时，必须额外检查：

- 是否把重计算放在请求路径上。
- 是否每次页面渲染都重复查数据库或调用外部 API。
- 是否缺少分页、缓存、去重、索引或批量查询。
- 是否服务端组件、SSR、server action、API route 和客户端请求职责混乱。
- 是否把大对象、大列表、大 JSON 直接传给前端。
- 是否因为部署在低配实例、冷启动、边缘/地区不匹配导致偶发卡顿。
- 是否在客户端重复 hydration、重复请求、重复状态计算。

全栈一体只有在边界清楚、数据路径可控、性能预算可验证时才允许作为默认推荐。

## 常见卡顿来源

AI 在排查或开发时必须优先检查：

- 无分页列表、一次性加载全部数据。
- N+1 查询、缺少索引、重复数据库请求。
- 请求瀑布流、重复请求、没有缓存或请求去重。
- 大组件重复渲染、无 memo、状态放置过高。
- 大包体积、组件库全量引入、图表/编辑器未懒加载。
- 图片未压缩、未设置尺寸、缺少 lazy loading。
- 同步计算、JSON 解析、加解密、格式化在主线程阻塞。
- 日志、埋点、AI 调用、第三方 SDK 阻塞核心路径。
- 开发环境和生产环境差异导致误判。

## 阶段门禁

### Architecture 阶段

- 说明核心路径性能预算。
- 说明全栈一体、前后端分离、静态站、API 服务等交付形态的性能取舍。
- 说明数据读取、缓存、分页、索引、异步任务和部署区域策略。

### Contract 阶段

- API 必须定义分页、排序、过滤、字段选择、错误码和超时策略。
- 大列表、大文件、导入导出、AI 任务必须定义异步或分批策略。

### Build 阶段

- 前端必须处理 loading、skeleton、空状态、错误状态和重试。
- 后端必须避免 N+1 查询，关键查询必须有索引或解释。
- AI/第三方调用必须有超时、重试、限流、熔断和降级。

### QA/Security 阶段

- 必须验证核心路径无明显卡顿。
- Web 项目必须尽量运行 Lighthouse、Playwright、框架 build analyzer、Web Vitals 或等价检查。
- 后端/API 项目必须至少记录关键接口耗时和慢查询风险。

## 性能验收输出

每次涉及核心路径、列表页、图表、文件、AI、数据库或全栈渲染时，必须输出：

```markdown
## 性能检查

- 核心路径：
- 性能预算：
- 已验证项：
- 发现的瓶颈：
- 优化措施：
- 未覆盖风险：
- 是否阻塞发布：
```

没有运行自动化性能工具时，必须说明原因，并至少做手工核心路径检查和代码级瓶颈排查。
