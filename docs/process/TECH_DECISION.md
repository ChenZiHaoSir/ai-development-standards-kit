# 技术选型决策规范

技术选型不是“哪个最快就用哪个”。AI 在选择框架、数据库、组件库、状态管理、ORM、AI SDK、部署平台、测试工具或关键依赖前，必须按本文件执行。

## 决策原则

默认优先级：

1. 安全性：认证、授权、数据保护、供应链风险、漏洞响应。
2. 可维护性：团队可理解、文档完整、社区活跃、升级路径清晰。
3. 稳定性：长期维护、版本成熟、生产案例充分。
4. 性能：满足核心路径、可观测、可优化、不会制造明显瓶颈。
5. 生态兼容：与项目语言、框架、部署、测试和 CI 兼容。
6. 交付效率：在不牺牲以上底线的前提下提高开发速度。
7. 可替换性：封装边界清晰，未来可以迁移或替换。

交付速度不能排在安全、性能、维护性之前。

## 强制选型门禁

涉及以下内容时，必须先写选型说明或 ADR：

- 前端框架、后端框架、移动端/桌面端框架。
- 数据库、缓存、消息队列、搜索引擎、对象存储。
- ORM、鉴权方案、权限模型、支付、文件上传、AI 模型调用 SDK。
- UI 组件库、状态管理、表单方案、请求库。
- 部署平台、CI/CD、日志、监控、错误追踪。
- 测试框架、E2E 工具、性能测试和安全扫描工具。
- 任何会影响长期架构、数据安全、团队协作或迁移成本的依赖。

## 选型评分表

AI 必须至少比较“推荐方案、备选方案、不采用方案”三类选择。

| 维度 | 权重 | 推荐方案 | 备选方案 | 不采用方案 |
| --- | --- | --- | --- | --- |
| 安全性 | 20 |  |  |  |
| 可维护性 | 20 |  |  |  |
| 稳定性 | 15 |  |  |  |
| 性能 | 15 |  |  |  |
| 生态兼容 | 10 |  |  |  |
| 交付效率 | 10 |  |  |  |
| 可替换性 | 10 |  |  |  |

评分低于 70 分的方案不得作为默认选型。安全性低于 15 分或可维护性低于 15 分的方案不得采用，除非用户明确接受风险并记录补偿措施。

## ADR 模板

新增或改变关键技术选型时，在 `docs/architecture/decisions/` 下创建 ADR：

```text

## 目标与非目标

## 退出策略

```

## 禁止事项

- 禁止只因为“最快能做完”而选择低安全、低维护、低稳定性的技术。
- 禁止引入无人维护、文档缺失、生态不兼容或许可证不清晰的依赖。
- 禁止在核心链路使用实验性工具，除非产品目标就是验证该工具。
- 禁止为了炫技选择复杂架构、冷门框架或团队不熟悉的工具。
- 禁止绕过官方脚手架、官方推荐配置和成熟模板去手搓项目底座。

## 官方优先

搭建项目框架时，优先使用官方脚手架或主流社区模板：

- Next.js：`create-next-app`
- Vite：`create-vite`
- NestJS：`@nestjs/cli`
- FastAPI：官方项目结构或成熟模板
- Spring Boot：Spring Initializr
- Tauri/Electron：官方 CLI

AI 不得在有官方脚手架时手写等价底座，除非用户明确要求或官方方案不能满足项目约束。
