# Observability Baseline

本文件定义日志、错误监控、性能监控、埋点统计和告警的最低要求。项目上线不是结束，必须能看见错误、性能、用户行为和业务指标。

## 核心原则

- 没有监控和埋点的功能不得直接全量发布。
- 监控优先覆盖核心路径、登录、支付、保存、导出、AI 调用、后台任务和数据写入。
- 日志、埋点和错误上报不得包含密钥、token、完整 prompt、身份证、手机号、邮箱、地址、银行卡或真实用户敏感内容。
- 埋点必须服务产品判断和问题定位，不得为了“多收集”无边界采集用户数据。

## 错误日志监控

必须明确：

- 前端错误采集方式：运行时异常、接口错误、资源加载失败。
- 后端错误采集方式：异常日志、请求 ID、错误码、耗时、调用链。
- 日志脱敏规则：敏感字段屏蔽、prompt 摘要、用户标识匿名化。
- 查询方式：如何按时间、用户、请求 ID、错误码定位问题。
- P0/P1 告警条件：应用不可用、核心流程失败、错误率异常、数据写入失败。

## 性能监控

必须覆盖：

- 首屏、路由切换、核心交互响应、接口 P95、数据库慢查询。
- 文件上传下载、图表渲染、列表分页、AI 调用耗时。
- 前端长任务、重复请求、请求瀑布和资源体积。
- 发布前后性能对比，发现回退必须记录原因和处理方案。

详细预算见 `docs/process/PERFORMANCE_BASELINE.md`。

## 埋点统计

必须先输出埋点方案：

```markdown
## 埋点方案

- 核心业务目标：
- 核心路径：
- 事件命名规则：
- 必须事件：
- 事件字段：
- 不允许采集字段：
- 数据口径：
- 验证方式：
```

事件命名建议：

```text
模块_对象_动作
```

示例：

```text
auth_login_submit
project_create_success
order_pay_failed
ai_generate_retry
```

每个事件必须说明：

- 触发时机。
- 字段含义和类型。
- 是否包含用户标识。
- 是否可用于漏斗、留存、转化或异常定位。
- 如何在测试环境验证事件是否上报。

## 告警和发布观察

发布前必须确认：

- 错误率阈值。
- 延迟阈值。
- 核心业务指标阈值。
- 告警接收人或处理方式。
- 回滚触发条件。
- 灰度观察时长。

## 禁止行为

- 禁止把完整日志、完整用户数据、完整 prompt 或敏感字段发到第三方监控。
- 禁止只有 console.log，没有结构化日志和定位字段。
- 禁止上线后才临时补错误监控和核心埋点。
- 禁止未定义回滚触发条件就灰度或全量发布。
