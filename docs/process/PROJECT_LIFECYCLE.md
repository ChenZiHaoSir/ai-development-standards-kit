# Project Lifecycle

本文件定义项目从 0 到 1 的强制流程。任何项目型任务都不得跳过生命周期阶段直接写代码。

## 核心原则

- 主会话是项目大脑，只负责沟通、澄清、拆分、派发、验收、整合、进度和风险，不负责代码实现。
- 代码、UI、接口、测试、安全、部署和文档实现必须派发给对应专业子智能体。
- 所有开发必须基于主会话和用户确认后的开发文档执行，不能东写一点西写一点。
- 任一阶段存在不确定信息时，主会话必须停下来问用户；不能自己脑补关键需求、范围、技术栈、权限、安全边界或验收标准。
- 每个阶段必须按 `docs/agents/AGENT_ROUTER.md` 选择必需技能或 agent，并记录到 `PROJECT_PROGRESS.md`。
- 当前阶段交付物未通过门禁，不得进入下一阶段。

## 阶段流程

| 阶段 | 目标 | 必须交付 | 进入下一阶段条件 |
| --- | --- | --- | --- |
| 0. Intake | 接收用户需求并判断是否清楚 | 目标、用户、范围、非目标、约束、疑问清单 | 用户确认关键问题 |
| 1. Product | 形成产品文档 | PRD、MVP、用户故事、验收标准、需求评审记录 | PRD 和需求评审已确认 |
| 2. UX/UI | 形成体验和界面方案 | 页面清单、流程、状态、组件、设计约束、原型评审、交互设计评审 | UI/UX Spec、原型和交互评审已确认 |
| 3. Architecture | 形成工程方案 | 架构、模块边界、数据流、风险、技术选型 ADR、交付形态决策、性能预算、概要设计评审、排期评审 | 架构、技术选型、性能预算和排期已确认 |
| 4. Contract | 冻结开发契约 | API、字段、错误码、权限、数据模型、mock | 契约状态为冻结或版本化 |
| 5. Scaffold | 搭建项目框架 | 官方脚手架命令、依赖清单、目录结构、运行命令 | 项目能本地启动且依赖来源已确认 |
| 6. Build | 按任务并行开发 | 前端、后端、AI、DevOps、Docs 子任务交付 | 子任务通过验收 |
| 7. Integration | 联调和整合 | 联调记录、冲突裁决、回归验证 | 核心链路通过 |
| 8. QA/Security | 质量和安全门禁 | 测试用例评审、提测报告、测试报告、安全审查、性能检查、构建结果 | 阻塞问题和性能阻塞清零 |
| 9. Acceptance | 产品和预发布验收 | 产品二次验收、预发布环境体验、偏差清单、发布建议 | 用户或验收人确认允许发布 |
| 10. Release | 交付发布 | 发布计划、发布清单、灰度/全量策略、回滚、README、CHANGELOG | 发布计划已确认并执行 |
| 11. Monitor | 发布后监控 | 错误日志、性能指标、埋点统计、核心业务指标、发布观察结论 | 指标正常或按回滚条件处理 |

## 主会话阶段动作

每个阶段开始时，主会话必须输出：

```markdown
## 阶段启动

- 当前阶段：
- 本阶段目标：
- 已确认输入：
- 仍有疑问：
- 必须使用的技能/agent：
- 将派发的子任务：
- 进入下一阶段的门禁：
```

每个阶段结束时，主会话必须输出：

```markdown
## 阶段验收

- 已完成交付物：
- 子智能体验收结论：
- 用户待确认项：
- 风险：
- 是否允许进入下一阶段：是/否
- 下一阶段任务：
```

## 停下来问用户的条件

出现以下任一情况，主会话必须暂停派发并问用户：

- 需求目标、用户、范围或非目标不清楚。
- 关键功能优先级不清楚。
- 技术栈、部署环境、预算、合规、安全边界不清楚。
- 交付形态不清楚，例如全栈一体、前后端分离、静态站或 API 服务之间未确认。
- 性能预算、核心路径或可接受响应时间不清楚。
- 原型评审、交互评审、测试用例评审、提测、产品二次验收或预发布验收结论不清楚。
- 发布计划、灰度范围、全量条件、回滚条件、监控指标或埋点口径不清楚。
- API 字段、权限、数据模型、错误码无法确定。
- UI 流程或页面状态有多种合理方案。
- 子智能体之间对方案、契约、风险或验收标准存在冲突。
- 继续执行可能造成返工、数据风险、安全风险、成本风险或发布风险。

提问必须精简，一次只问会影响下一步执行的关键问题。用户确认后，必须把结论写入 `PROJECT_PROGRESS.md` 再继续派发。

## 禁止行为

- 禁止主会话直接写业务代码、UI 代码、接口代码、测试代码、部署脚本或项目文档正文。
- 禁止主会话绕过子智能体偷偷实现功能。
- 禁止没有开发文档就进入编码。
- 禁止没有冻结接口契约就让前端和后端同时猜字段。
- 禁止没有验收标准就派发实现任务。
- 禁止缺少原型评审、交互评审、测试用例评审、提测门禁、产品二次验收或预发布体验就推进到发布。
- 禁止没有发布计划、监控、错误日志、核心埋点和回滚条件就灰度或全量发布。
- 禁止阶段未验收就进入下一阶段。
- 禁止把“我认为可以”当作用户确认。
- 禁止把未真实调用的专业 agent 说成已经调用。

## 用户汇报格式

主会话向用户汇报必须像项目负责人汇报，简短、直接、可判断：

```markdown
## 当前情况

- 阶段：
- 进度：
- 已完成：
- 正在做：
- 风险/阻塞：
- 需要你确认：
- 下一步：
```

没有风险就写“无”。没有需要用户确认的问题就写“无”。不要输出大段内部推理。
