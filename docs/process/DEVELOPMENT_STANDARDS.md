# Development Standards

这是新项目默认研发规范。项目可以补充分端细则，但不得削弱质量、安全、测试和交付底线。

## 1. 项目原则

- 所有功能必须服务核心用户路径。
- MVP 只做一个可闭环、可验证、可交付的主流程。
- 优先采用成熟大厂或主流社区长期维护的技术栈，不自造基础框架。
- 产品、架构、测试、发布和安全文档必须随实现同步更新。
- 完成标准是用户能稳定完成目标，不是功能刚好能跑。
- 默认采用最小必要实现：先判断是否需要做，再优先使用标准库、平台原生能力和已有依赖，最后才写自定义代码。

## 2. 技术选型

- 前端：React + TypeScript + Next.js/Vite，或项目明确认可的主流框架。
- 管理后台：React + TypeScript + Ant Design/Arco/MUI，统一表格、表单、权限和路由。
- Node.js：NestJS/Fastify/Express，统一 DTO、校验、异常、日志和配置。
- Java：Spring Boot/Spring Cloud，遵守 Controller-Service-Repository 分层。
- Python：FastAPI/Django，统一 schema、依赖注入、迁移、任务队列和测试。
- 桌面端：Tauri/Electron，系统能力封装在桌面层。
- AI 服务：Python + FastAPI/Ray/Celery/LangGraph 等成熟组件，模型调用集中封装。

非主流技术必须在架构文档中说明选择理由、替代方案、风险和退出策略。

## 3. 最小必要实现

实现前按顺序判断：

1. 这个功能是否真的需要存在。
2. 标准库是否已经提供。
3. 浏览器、操作系统、数据库或云平台原生能力是否已经覆盖。
4. 项目已安装依赖是否已经解决。
5. 是否可以用一两行清晰代码完成。
6. 只有以上都不成立时，才写最小可工作的实现。

禁止为一个实现创建接口、工厂、抽象基类或插件系统；禁止为“以后可能需要”提前搭脚手架；禁止为小问题新增依赖。

最小实现不能省掉输入校验、数据保护、权限、安全、可访问性、错误处理和必要验证。详细规则见 `docs/process/MINIMAL_IMPLEMENTATION.md`。

## 4. 架构边界

推荐分层：

| 层级 | 职责 |
| --- | --- |
| 表现层 | 页面、组件、交互、展示状态 |
| 应用层 | 用户流程、业务编排、状态组合 |
| 领域层 | 核心模型、规则、纯逻辑 |
| 基础设施层 | 数据库、文件、网络、第三方服务、系统能力 |
| 通用层 | UI、格式化、日志、错误、配置、校验 |

规则：

- 依赖方向必须单向，不允许循环依赖。
- 表现层不得直接访问数据库、文件系统、第三方 SDK、密钥和系统能力。
- 领域层不得依赖 UI 框架、HTTP 框架、数据库 SDK 或云厂商 SDK。
- 外部服务调用必须经过 adapter/client 封装。
- 相同逻辑出现 2 次及以上必须抽离。
- 可调参数必须进入配置层、常量层或项目约定位置。

## 5. 复用与重复逻辑治理

重复功能和相似逻辑必须主动治理，不能靠复制粘贴扩展项目。

- 相同业务逻辑出现 2 次及以上，必须抽离为通用函数、hook、service、domain rule、adapter 或组件。
- 相似逻辑仅参数、字段、文案、状态或策略不同，应优先改造成可配置、可组合或策略化实现。
- 认证、权限、日志、错误处理、表单校验、数据请求、缓存、AI 调用、SDK 初始化、脱敏、分页、重试和限流必须集中封装。
- 同类 UI 控件、表格、弹窗、表单、上传、导出和状态展示必须沉淀为统一组件或业务组件，禁止每个页面重新实现一套。
- 抽离复用时必须保持边界清晰：领域规则放领域层，外部调用放基础设施层，展示复用放组件层，流程编排放应用层。
- 不为了“看起来复用”制造过度抽象；只有真实重复、稳定变化点或明确复用场景时才抽象。
- 每次新增相似功能前，必须先搜索项目中是否已有可复用实现；如果不用，需说明原因。
- 公共函数必须有清晰命名、输入输出类型、边界处理和必要测试，不能变成无边界的 `utils` 垃圾桶。

## 6. 数据与安全

- 先做数据分级，再决定存储、传输和日志策略。
- 密钥不得写入源码、日志、测试输出、截图或提交记录。
- 高敏感数据不得通过 URL query、referer 或普通日志传播。
- 日志只记录排障必要信息，如任务 ID、错误码、耗时和脱敏状态。
- 服务端必须校验接口级、数据级和操作级权限。
- 管理员权限必须最小化，不能作为普通业务流程兜底。

## 7. AI 功能规范

- AI 调用必须由用户显式触发，除非产品文档明确说明并获得授权。
- Prompt 必须模板化、版本化，禁止散落在业务代码中。
- 上下文打包、模型调用、结果解析和错误处理必须集中封装。
- AI 输出不得直接覆盖用户原始内容，必须进入建议区、草稿区或可审阅状态。
- 模型调用必须有超时、重试、并发限制、限流、熔断和降级方案。
- 记录供应商、模型名、参数、模板版本、上下文清单、输出版本和错误信息，但不得记录密钥或完整敏感正文。

## 8. 测试策略

测试优先保护三类风险：

1. 用户数据丢失。
2. 敏感信息泄露。
3. 核心路径不可用。

按风险选择测试层级：

- 单元测试：纯逻辑、转换、边界、格式化、校验。
- 集成测试：数据访问、服务编排、外部接口 mock、导入导出。
- E2E：首次使用、核心业务路径、错误恢复、关键设置。
- Smoke：启动、主页面、关键按钮、关键窗口尺寸、构建产物基本可用。

## 9. 质量门禁

PR 合并前默认强制：

- install / lockfile 校验
- lint
- typecheck
- test
- build
- 密钥扫描
- 依赖漏洞扫描

涉及接口契约、数据库迁移、UI、安全、性能或发布流程时，必须追加对应专项验证。

## 10. 完成定义

一个功能只有同时满足以下条件才算完成：

- 验收标准已满足。
- 核心逻辑有测试或明确验证记录。
- 错误状态和边界情况已处理。
- 不泄露敏感信息。
- 不破坏核心路径。
- 性能影响已评估。
- 相关文档已同步。
- 工作区没有无关变更、临时文件、密钥或真实用户数据。
