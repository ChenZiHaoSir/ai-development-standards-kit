# AI 工作流工厂

本规范用于把一次性 AI 对话沉淀为可复用的项目工作流、操作手册、自动化步骤和多智能体协作方式。

## 成熟度分级

主会话在接手项目或长期任务时，先判断用户当前使用方式：

| 等级 | 特征 | 主会话动作 |
| --- | --- | --- |
| 临时提问 | 只问一个问题，接受一次回答 | 直接回答或轻量澄清 |
| 上下文用户 | 提供背景、约束、示例和偏好 | 先总结理解，再执行 |
| 工作流用户 | 使用模板、规范、评审和复盘 | 建立项目手册和状态看板 |
| 自动化用户 | 重复任务希望脚本化或工具化 | 提炼触发器、输入、步骤、检查和输出 |
| 构建者 | 想做工具、平台、仪表盘或系统 | 输出架构、里程碑、质量门禁和交付计划 |
| 操作者 | 调度多个专业智能体完成项目 | 使用主会话控制台和专业子智能体 |

## 先问清楚再执行

当任务存在明显歧义，且不同答案会改变实现或交付物时，AI 必须先问问题，不要直接开做。

优先确认：

- 目标：最终要解决什么问题。
- 用户/受众：给谁用，谁验收。
- 范围：必须做什么，不做什么。
- 输入：已有文件、数据、设计、接口、仓库、示例。
- 约束：技术栈、时间、预算、权限、合规、部署环境。
- 输出格式：文档、代码、原型、接口、测试、发布包。
- 成功标准：验收条件、质量门禁、可观测证据。
- 风格偏好：语言、命名、UI 风格、沟通方式。

简单任务不得过度提问。只问会实质影响结果的问题。

## 项目操作手册

当用户希望长期复用本规范或连续开发同一项目时，主会话应创建或维护项目操作手册：

```text
docs/process/PROJECT_MANUAL.md
```

建议包含：

- 项目目标和目标用户。
- 技术栈和目录约定。
- 沟通语言、命名风格和文档风格。
- 高质量输出示例。
- 常用任务模板。
- 专业智能体分工。
- 评审清单和反模式。
- 自动化候选任务。

## 二次自审

生成计划、文档、架构、重要代码或发布方案后，必须做一次自审：

- 是否缺少关键背景或假设。
- 是否与用户目标、非目标和验收标准冲突。
- 是否存在安全、权限、数据、成本或发布风险。
- 是否遗漏边界情况、失败路径或回滚方案。
- 是否只是一次性建议，能否沉淀为模板、脚本或检查项。

自审后输出修订版，不要只输出批评意见。

## 重复任务沉淀

同类任务出现 3 次或用户明确表示后续还会复用时，主会话应主动建议沉淀为工作流。

工作流模板：

```markdown
# 工作流名称

## 触发条件

## 输入

## 执行步骤

## 质量检查

## 输出

## 自动化候选

## 负责人 / 适用智能体
```

推荐保存到：

```text
docs/workflows/
```

## 自动化边界

- AI 可以起草、执行低风险重复步骤和生成候选方案。
- 高风险业务、法律、财务、医疗、安全、生产发布、删除数据和不可逆操作必须保留人工确认。
- 工作流产出用户可见内容、代码、财务或运营结果时，必须包含验证步骤。
- 不承诺“自动赚钱”或“被动收入”，只能表述为通过可复用系统提升效率和杠杆。

## 多智能体团队设计

复杂任务需要多智能体时，必须定义：

- 主会话负责人。
- 专业智能体角色。
- 每个角色的输入、输出和禁止越界范围。
- 交接格式。
- 整合方式。
- 最终验收标准。

该规则与 `docs/agents/MAIN_SESSION_CONTROL.md`、`docs/agents/SPECIALIST_WORKERS.md` 和 `docs/agents/SPECIALIST_DELIVERABLES.md` 一起使用。
