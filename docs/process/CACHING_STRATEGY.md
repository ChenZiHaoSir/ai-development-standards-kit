# 缓存策略

本文件定义何时使用缓存、缓存层级、失效策略和安全防护。

## 何时使用缓存

以下场景必须考虑缓存：

- 数据库查询频率高但变化频率低（配置数据、字典表）
- 接口响应计算成本高（复杂聚合、AI 推理）
- 高并发读场景（商品详情、文章页面）
- 用户会话、Token 校验
- 第三方 API 调用（有频率限制或按量计费）

以下场景不使用缓存：

- 实时性要求极高的数据（金融行情、订单状态）
- 单次查询、开发环境内部接口
- 数据敏感且不能有延迟（权限校验、敏感操作确认）

## 缓存层级

| 层级 | 实现方式 | 适用场景 | 失效策略 |
| --- | --- | --- | --- |
| 浏览器缓存 | HTTP Cache、Service Worker | 静态资源、公开页面 | 版本化文件名、ETag |
| CDN 缓存 | CDN 边缘节点 | 静态资源、公开 API | TTL、主动刷新 |
| 应用层缓存 | 进程内存缓存（如 LRU Cache） | 高频小对象 | TTL、数量限制 |
| 分布式缓存 | Redis/Memcached | 热点数据、会话、排行 | TTL、主动失效、消息通知 |
| 数据库缓存 | MySQL Query Cache、Buffer Pool | 查询结果 | 自动失效 |

## 缓存命名规范

```
<业务>:<模块>:<类型>:<标识>
```

示例：

```
user:123:profile
product:456:detail
order:789:list
config:system:all
rate:login:ip:1.2.3.4
```

## 失效策略

### TTL 失效

- 配置类数据：24 小时（变更时主动失效）
- 用户资料：1 小时（修改时主动失效）
- 商品信息：30 分钟（变更时主动失效）
- 会话数据：Session 超时时间

### 主动失效

- 数据写入后立即删除缓存（先删缓存后写库，或使用延迟双删）
- 支持消息队列通知缓存失效（适合多实例部署）

### 版本化失效

- 静态资源文件名添加版本号：`app.abc123.js`
- API 响应添加 ETag，客户端条件请求

## 缓存安全防护

### 缓存击穿

防护方案：

- 互斥锁（分布式锁）：高并发时只允许一个请求查库
- 热点数据永不过期，异步更新
- 布隆过滤器拦截不存在的 Key

### 缓存雪崩

防护方案：

- TTL 添加随机扰动，避免同时失效
- 熔断降级：缓存不可用时直接查询数据库
- 数据库连接池限流，防止被打垮

### 缓存穿透

防护方案：

- 布隆过滤器：拦截不存在的 Key
- 缓存空值：不存在的数据也缓存，TTL 较短（60s）
- 参数校验：拦截明显非法的查询参数

## 多级缓存设计

```
请求 -> 浏览器缓存 -> CDN -> 应用缓存 -> Redis -> 数据库
```

每增加一级缓存，必须评估：

- 缓存命中率预期（命中率 < 50% 时收益存疑）
- 缓存一致性风险（延迟窗口可接受）
- 缓存成本（内存、网络、维护复杂度）

## 禁止行为

- 禁止缓存用户敏感数据（密码、token、银行卡号）
- 禁止无 TTL 的缓存（必须有上限或异步更新机制）
- 禁止缓存异常响应（失败时不写入缓存）
- 禁止大量相同 Key 并发写入缓存（必须加锁或合并）
- 禁止缓存整个数据库结果集（必须有大小限制）
- 禁止在本地缓存敏感数据（必须有过期机制）
