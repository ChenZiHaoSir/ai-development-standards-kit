# 错误处理规范

本文件定义前端错误边界、后端全局异常处理、用户反馈和错误恢复的强制要求。

## 核心原则

- 所有错误必须有用户可见的反馈，不能"静默失败"。
- 错误信息必须对用户友好，不暴露技术细节。
- 错误发生后必须有恢复路径或重试引导。
- 错误日志必须结构化、可检索，支持按错误码定位。

## 前端错误处理

### Error Boundary

- 每个页面必须有 Error Boundary 组件捕获渲染错误。
- Error Boundary 必须显示友好错误页面，而非白屏。
- Error Boundary 必须提供重试按钮和返回首页链接。

### API 调用错误

```typescript
// 必须统一处理的错误类型
type AppError = {
  code: number;      // 业务错误码
  message: string;   // 用户可见错误信息
  requestId: string; // 用于日志查询
  detail?: string;   // 技术详情（仅开发环境显示）
};
```

错误处理策略：

| 错误类型 | 用户反馈 | 恢复方式 |
| --- | --- | --- |
| 网络错误 | "网络异常，请检查网络" | 重试按钮 |
| token 过期 | 跳转登录页 | 重新登录 |
| 参数错误 | 表单行内错误提示 | 用户修改输入 |
| 业务错误 | Toast/弹窗说明 | 用户调整操作 |
| 服务错误 | "服务繁忙，请稍后重试" | 自动重试 + 重试按钮 |
| 权限不足 | "无权限访问" | 引导申请权限 |

### Toast 通知

- 成功提示：3 秒后自动消失
- 警告提示：5 秒后自动消失
- 错误提示：需手动关闭，不自动消失
- 同一位置最多显示 3 条 Toast，超出后替换最早的

### 表单校验错误

- 实时校验：输入失焦时校验单字段
- 提交校验：提交时全量校验所有字段
- 错误信息显示在字段下方，不使用 alert
- 第一个错误字段自动聚焦

## 后端错误处理

### 全局异常处理器

后端必须实现全局异常处理器，统一捕获所有未处理异常：

| 异常类型 | HTTP 状态 | code | 日志级别 | 用户信息 |
| --- | --- | --- | --- | --- |
| 参数校验异常 | 400 | 3001 | WARN | 校验失败详情 |
| 认证异常 | 401 | 2001 | INFO | 未登录 |
| 权限异常 | 403 | 2003 | WARN | 无权限 |
| 资源不存在 | 404 | 4001 | INFO | 资源不存在 |
| 业务异常 | 400 | 4xxx | WARN | 业务规则说明 |
| 并发冲突 | 409 | 4005 | INFO | 资源被修改，请刷新 |
| 限流异常 | 429 | 1003 | WARN | 请求过于频繁 |
| 服务器异常 | 500 | 1001 | ERROR | 服务异常，请重试 |

### 异常处理规则

- 所有异常必须有全局处理器，不允许遗漏
- 数据库异常必须转换为业务错误码，不暴露 SQL 和表结构
- 第三方异常必须转换为业务错误码，不暴露第三方错误信息
- 错误日志必须包含 requestId、用户ID（脱敏）、请求参数摘要、异常堆栈
- 敏感数据必须脱敏后再记录日志

## 重试机制

| 场景 | 重试次数 | 间隔策略 | 说明 |
| --- | --- | --- | --- |
| 网络超时 | 2次 | 指数退避（1s, 2s） | 自动重试 |
| 5xx 服务错误 | 1次 | 指数退避（2s） | 自动重试 |
| 限流 | 0次 | 用户手动重试 | 禁止自动重试 |
| 401 未登录 | 0次 | 刷新 token 或跳转登录 | 禁止自动重试 |
| 403 无权限 | 0次 | 用户申请权限 | 禁止自动重试 |
| 400 参数错误 | 0次 | 用户修改输入 | 禁止自动重试 |
| POST/PUT 非幂等 | 0次 | 禁止自动重试 | 无幂等保证时禁止重试 |

## 错误监控

- 前端运行时异常必须上报到错误监控服务（如 Sentry、阿里云 ARMS）
- 上报内容：错误消息、堆栈、用户环境（OS、浏览器）、操作路径、请求链路
- P0 级错误（如支付失败、数据丢失）必须实时告警
- 所有错误必须关联 requestId，支持按请求链路追踪

## 禁止行为

- 禁止 `catch(e) {}` 空 catch 吞掉异常
- 禁止前端直接 `alert` 或 `console.log` 作为用户反馈
- 禁止后端直接返回 `500 Internal Server Error` 作为业务错误
- 禁止在 catch 中再次抛出未处理的异常
- 禁止错误提示中包含技术细节（如 SQL 语句、堆栈信息）
- 禁止前端对所有错误都做统一处理，必须按错误类型分策略
- 禁止错误发生后不记录日志
