# Code Review 规范

本文件定义 Code Review 的流程、职责分工和质量门禁。

## 核心原则

- Code Review 是质量门禁，不是形式流程。
- 所有代码必须经过 Review 才能合并到 main 分支。
- Review 发现的问题必须记录，不能口头说明后跳过。

## 职责分工

| 角色 | 职责 |
| --- | --- |
| 作者（PR 提交者） | 准备完整 PR 信息、主动说明风险点、响应 Review 意见 |
| Reviewer | 检查代码质量、安全性、规范符合性、提出改进建议 |
| Maintainer | 裁决争议、确认合并条件、确保整体一致性 |

## Review 触发条件

| 变更类型 | 要求 Review |
| --- | --- |
| 功能代码变更 | 必须 1 人 Review |
| 安全/权限相关变更 | 必须 2 人 Review |
| 架构/数据库/发布流程变更 | 必须 1 人 Review + 负责人审批 |
| 文档/配置变更 | 必须 1 人 Review |
| CI/CD 流水线变更 | 必须 1 人 Review |
| 依赖升级（patch/minor） | 必须 1 人 Review |
| 依赖升级（major） | 必须 1 人 Review + 负责人审批 |

## Review Checklist

### 功能正确性

- [ ] 代码实现了 PR 描述的业务目标
- [ ] 边界情况已处理（空值、超大值、并发、权限不足）
- [ ] 错误处理完整（用户可见反馈 + 日志记录）
- [ ] 状态处理正确（成功/失败/加载/空状态）

### 安全性

- [ ] 没有硬编码的密钥、密码、token
- [ ] 敏感数据已脱敏（日志、响应、截图）
- [ ] 权限校验已覆盖（接口级 + 数据级）
- [ ] SQL 查询无注入风险
- [ ] 文件上传有大小和类型限制
- [ ] 用户输入已校验和转义

### 性能

- [ ] 没有 N+1 查询
- [ ] 列表查询有分页、有索引
- [ ] 大对象未重复创建和序列化
- [ ] 组件渲染无明显性能问题
- [ ] 外部调用有超时

### 代码质量

- [ ] 函数/类职责单一，不过度复杂
- [ ] 命名清晰，注释说明"做什么"而非"怎么做"
- [ ] 没有重复代码（超过 3 行相似必须考虑抽取）
- [ ] 没有未使用的导入、变量、函数
- [ ] 没有 TODO 未关联 issue

### 规范符合性

- [ ] 符合项目编码规范（ESLint/Prettier/Style Guide）
- [ ] 符合项目架构分层约束
- [ ] 无循环依赖
- [ ] 公共函数有类型定义

## Review 响应时间

| 优先级 | Review 响应时间 | 说明 |
| --- | --- | --- |
| P0（紧急修复） | 2 小时内 | 生产故障、安全漏洞 |
| P1（功能开发） | 4 小时内 | 核心功能开发 |
| P2（优化重构） | 8 小时内 | 重构、文档、配置 |

## Review 结论

每次 Review 必须有明确结论：

- **Approved**：可以合并，无阻塞问题
- **Changes Requested**：需要修改后重新 Review
- **Comment**：有建议但不阻塞合并

## Review 结果处理

- Changes Requested 的 PR 必须在修改后重新请求 Review
- 作者回复每条 Review 意见时，必须说明"已修改"或提供修改方案
- 不同意的 Review 意见必须记录到 PR conversation，由 Maintainer 裁决

## 合并条件

PR 合并必须满足以下所有条件：

- CI/CD 全部通过（lint、typecheck、test、build）
- 至少 1 人 Approved（安全/权限变更需 2 人）
- 无未解决的 Changes Requested
- 分支是最新的 main（rebase 或 merge main 后 CI 通过）
- 密钥扫描通过
- 无破坏性变更未经负责人确认

## 禁止行为

- 禁止未 Review 就合并（紧急 P0 需要 Maintainer 特批并记录原因）
- 禁止只 Approve 不看代码
- 禁止在 Review 意见中使用模糊表述（"可能有问题"），必须指出具体位置和原因
- 禁止作者本人 Approve 自己的 PR
- 禁止把 Review 意见记在 PR 外部（口头沟通不记录）
