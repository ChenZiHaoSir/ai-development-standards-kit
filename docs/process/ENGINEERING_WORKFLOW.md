# Engineering Workflow

## 单功能开发闭环

1. 写清用户问题、业务目标和非目标。
2. 判断是否需要先问清楚上下文：目标、受众、范围、输入、约束、输出格式、成功标准。
3. 写清验收标准。
4. 说明涉及模块、输入输出、数据流和性能影响。
5. 执行最小必要实现判断：是否需要做、标准库/平台/已有依赖是否已覆盖、是否必须新增自定义代码。
6. 执行上下文预算判断：先搜索和摘要，再读取必要文件片段；长输出保存到文件并只引用关键证据。
7. 先定义契约，再实现模块。
8. 修改代码。
9. 补充单元测试、集成测试或 smoke 验证。
10. 运行必要验证命令。
11. 对计划、文档、架构、重要代码或发布方案做二次自审并修订。
12. 更新相关文档和进度记录。
13. 如果同类任务已经重复 3 次，沉淀为 `docs/workflows/` 下的可复用工作流。
14. 检查工作区状态。
15. 使用中文提交信息提交，并按项目要求推送。

## 上下文采集

复杂或模糊任务必须先收集上下文。只问会实质影响结果的问题，避免为了形式过度提问。

优先确认：

- 目标和非目标。
- 目标用户或验收人。
- 输入材料和已有约束。
- 输出格式和完成标准。
- 技术栈、部署、权限和安全边界。
- 示例、偏好和反模式。

详细规则见 `docs/process/AI_WORKFLOW_FACTORY.md`。

## 默认目录基线

```text
src/
  app/
  pages/
  features/
  shared/
  services/
  domain/
  infrastructure/
  test/
docs/
scripts/
```

框架有官方目录规范时，以官方规范优先；本文目录作为边界参考。

## 验证命令基线

```bash
pnpm typecheck
pnpm lint
pnpm test
pnpm smoke
pnpm benchmark
pnpm build
```

没有 `pnpm` 的项目应提供等价命令，并在 README 写清。

## 上下文预算

- 探索仓库优先使用 `rg --files`、`rg -n`、`git diff --stat`、`git diff --name-only`。
- 读取文件优先读取相关片段，不默认全量读取大文件。
- 测试、构建、lint 输出优先保留失败、警告和摘要。
- 如果环境安装了 `rtk`，可以优先使用 `rtk git status`、`rtk git diff`、`rtk read`、`rtk grep`、`rtk test` 等压缩输出命令。
- 详细规则见 `docs/process/CONTEXT_BUDGET.md`。

## PR 规则

- 一个 PR 只解决一个明确问题。
- 默认不超过 400 行有效代码变更，超过必须说明原因或拆分。
- 涉及 UI 必须提供截图、录屏或可复现验证证据。
- 高风险模块至少 2 名评审或负责人确认。
- CI 失败不得合并，除非有负责人批准的临时豁免和补偿任务。

## 工作流沉淀

- 重复出现 3 次以上的任务应沉淀为工作流。
- 工作流必须包含触发条件、输入、步骤、检查、输出和适用智能体。
- 可脚本化的低风险重复步骤应优先放入 `scripts/`。
- 高风险决策和不可逆操作不得完全自动化，必须保留人工确认。
