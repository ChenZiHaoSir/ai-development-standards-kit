# Minimal Implementation

本规范吸收 `ponytail` 的核心思想：少写代码不是偷懒，而是减少维护面、依赖面、缺陷面和协作成本。最小实现必须仍然满足安全、测试、可访问性、数据保护和用户目标。

## 实现前六级判断

写代码前按顺序停在第一个成立的选项：

1. **这个功能真的需要存在吗？** 如果只是猜测未来会用，先不做。
2. **标准库是否已经提供？** 优先使用语言标准库。
3. **平台原生能力是否已经覆盖？** 浏览器、操作系统、数据库、云平台已有能力优先。
4. **项目已安装依赖是否已经解决？** 优先复用已有依赖，不为小问题新增依赖。
5. **能否用一两行清晰代码解决？** 可以就不要建类、建层、建配置。
6. **只有以上都不成立时，才写最小可工作的实现。**

## 禁止过度工程

- 不为一个实现创建接口、工厂、抽象基类或插件系统。
- 不为一个不会变化的值创建配置项。
- 不为“以后可能需要”提前搭脚手架。
- 不新增没有明确收益的依赖。
- 不把简单逻辑拆成多层文件来制造架构感。
- 不在未证明需要前实现缓存、队列、复杂状态机、插件化、多租户扩展点或自定义 DSL。

## 优先平台能力

| 场景 | 优先方案 |
| --- | --- |
| 日期、时间、颜色、范围输入 | HTML 原生 input 类型 |
| 弹窗 | `<dialog>` 或项目已有组件 |
| 折叠面板 | `<details><summary>` 或项目已有组件 |
| 金额、数字、日期格式化 | `Intl.NumberFormat` / `Intl.DateTimeFormat` |
| URL query | `URLSearchParams` |
| 深拷贝 | `structuredClone` |
| UUID | `crypto.randomUUID()` 或后端/数据库原生 UUID |
| 简单去重 | `Set` |
| 文件和路径 | Node/Python 标准库 |
| 唯一性、范围、外键 | 数据库约束 |
| 聚合、排序、分页 | 数据库查询能力，必要时加索引 |

## 不能偷懒的地方

以下内容不能因为“最小实现”被省掉：

- 信任边界输入校验。
- 防止数据丢失的错误处理。
- 身份认证、授权和数据范围校验。
- 日志脱敏和密钥保护。
- 可访问性基础要求。
- 用户明确要求的功能。
- 涉及金钱、权限、数据迁移、安全、AI 输出的最小验证。

## 最小验证

非平凡逻辑必须留下一个最小可运行验证：

- 一个 assert 自检。
- 一个小单测。
- 一个 smoke 步骤。
- 一个可复现命令。

不要为了简单功能搭重型测试框架；但涉及分支、循环、解析、金额、安全、权限或数据写入时，不能没有验证。

## 审查清单

Review 时额外检查：

- 是否有手写标准库已有能力？
- 是否新增了不必要依赖？
- 是否有只有一个实现的抽象层？
- 是否有未来式配置、插件、扩展点？
- 是否有可以删除的样板代码？
- 是否能用已有组件、已有 service 或平台能力替代？
- 是否保留了必要的安全、错误处理、可访问性和验证？
