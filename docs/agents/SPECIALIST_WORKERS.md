# Specialist Workers

子智能体必须职责单一。每个后台子任务、后台 agent 或子会话只做自己的专业工作，不跨角色包办项目。

在 Codex 中，默认把专业智能体理解为后台执行单元，而不是用户可见的新窗口。主会话负责派发、跟踪、整合和汇报；用户不需要进入子任务窗口沟通。

如果用户明确要求可见子智能体窗口，则每个可见窗口就是一个专业执行单元。主会话只能编排、派发、验收和整合，不能绕过可见窗口直接承担专项实现。

## 默认专业角色

| 角色 | 只负责 | 不负责 |
| --- | --- | --- |
| Product / PM | 需求、范围、验收标准、任务拆分 | 直接写代码 |
| UI / UX | 信息架构、视觉、交互、设计系统、状态覆盖 | 后端接口、数据库、部署 |
| Frontend | 页面、组件、状态、路由、接口对接、前端测试 | 后端业务逻辑、数据库迁移 |
| Backend | API、服务、数据库、权限、事务、缓存、队列 | UI 视觉、前端组件 |
| AI Engineer | Prompt、模型调用、上下文打包、解析、限流、熔断、审计 | 普通 UI 细节、非 AI 后端通用代码 |
| QA / Testing | 测试计划、用例、自动化测试、复现、验证证据 | 直接改生产代码，除非明确要求 |
| Security | 密钥、权限、数据分级、日志脱敏、依赖风险、AI 安全 | 普通功能实现 |
| DevOps / SRE | CI/CD、环境、部署、监控、告警、回滚 | 产品需求和 UI 设计 |
| Docs | README、开发文档、发布说明、变更日志 | 功能实现和架构裁决 |

## 子智能体任务包

主会话分配任务时必须提供：

```markdown
## /goal

一句话说明本子智能体目标。

## 输入

- 相关文件：
- 需求或上游交付物：
- 接口契约或边界：

## 输出

- 必须产出的文件、说明或审查结论：

## 验收标准

- [ ] 

## 限制

- 只能修改：
- 不能修改：
- 不能跨越的职责边界：
```

## Codex 后台执行规则

- 优先使用 Codex 可用的后台子任务、后台 agent、多 agent 工具或内部委派能力。
- 不默认创建用户可见的新窗口、新会话或新线程。
- 后台子任务必须把结论、改动文件、验证结果、未完成项和风险回传给主会话。
- 主会话必须在 `PROJECT_PROGRESS.md` 或等价状态看板中记录后台子任务状态。
- 如果当前环境不支持后台子任务，主会话应串行执行或明确说明降级方式，不得要求用户手动协调多个窗口。

## 可见子窗口执行规则

- 只有用户明确要求时，才创建可见子智能体窗口。
- 每个可见窗口必须绑定一个单一专业角色和一个清晰 `/goal`。
- 主会话不得在可见窗口模式下直接替代 UI、前端、后端、测试、安全、DevOps 或文档窗口完成专项实现。
- 可见窗口交付失败时，主会话必须返工、改派、降级或向用户提问，不能私自接手实现。
- 主会话必须把每个可见窗口的状态、交付物、验收结果和下一步记录到状态看板。

## 专业交付物

每个子智能体必须产出自己的专业文档，不能只交代码或口头结论。

| 角色 | 必须交付 |
| --- | --- |
| Product / PM | PRD、MVP 范围、验收标准、非目标 |
| UI / UX | UI/UX 文档、页面状态、设计 token、组件清单 |
| Frontend | 前端集成文档、路由、组件结构、状态模型、API 依赖 |
| Backend | API 文档、数据模型、错误码、权限、迁移说明 |
| AI Engineer | Prompt 文档、上下文字段、输出 schema、审计和降级 |
| QA / Testing | 测试计划、验收矩阵、验证命令、缺陷清单 |
| Security | 数据分级、权限矩阵、日志脱敏和高风险操作清单 |
| DevOps / SRE | 环境、配置、CI/CD、部署、监控、回滚 |
| Docs | README、使用说明、CHANGELOG、发布说明 |

详细模板见 `docs/agents/SPECIALIST_DELIVERABLES.md`。

## 职责边界规则

- UI 子智能体不能顺手实现后端。
- 前端子智能体不能顺手改数据库 schema。
- 后端子智能体不能顺手重做 UI。
- QA 子智能体优先报告和验证，不默认修改业务实现。
- 安全子智能体可以提出阻断意见，但最终裁决由主会话汇总后执行。
- 跨角色变更必须回到主会话重新拆任务。

## 联调沟通规则

子智能体可以向其他专业角色提出联调请求，但不能私下直接协商并修改契约。

- 前端发现接口字段、错误码、权限或数据格式问题时，向主会话提交联调请求。
- 后端发现前端调用方式、参数、状态处理或 mock 不一致时，向主会话提交联调请求。
- QA 发现跨端不一致时，向主会话提交缺陷或联调请求。
- 主会话负责登记、转发、裁决和更新状态看板。
- 被请求方只处理主会话明确分配的联调任务。

联调请求格式：

```markdown
## 联调请求

- 提出方：
- 请求处理方：
- 问题：
- 影响范围：
- 相关契约 / 文件：
- 阻塞程度：
- 期望输出：
- 建议验收：
```
