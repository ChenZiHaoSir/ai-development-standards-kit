# Parallel Development

并行开发的目标是提高效率，同时避免文件冲突、接口漂移和重复实现。

## 可以并行的前提

- 需求范围和非目标已明确。
- 接口契约、数据模型、错误码和权限边界已冻结或有版本标记。
- 每个子智能体的可修改文件范围已明确。
- 公共组件、通用函数和共享类型的归属已明确。
- 主会话负责整合，不让子智能体互相覆盖。
- 跨智能体联调请求必须通过主会话登记和转发。

## 默认并行拆分

| 工作流 | 可并行条件 | 典型交付物 |
| --- | --- | --- |
| UI / UX | 用户路径和核心页面已明确 | wireframe、交互状态、设计 token、组件清单 |
| Frontend | API 契约或 mock 已明确 | 页面、组件、状态、前端测试 |
| Backend | 数据模型和接口契约已明确 | API、service、repository、migration、后端测试 |
| QA | 验收标准已明确 | 测试计划、用例、自动化验证、缺陷清单 |
| Security | 数据流和权限模型已明确 | 风险清单、阻塞问题、修复建议 |
| DevOps | 运行命令和环境需求已明确 | CI、部署脚本、监控、回滚说明 |
| Docs | 目标和实际实现已明确 | README、变更日志、使用说明、发布说明 |

## 不允许并行的情况

- 多个智能体需要同时编辑同一文件。
- 接口契约未定，前端和后端都在猜字段。
- 数据库 schema 未定，但后端、测试和文档已开始写死结构。
- 公共组件或工具函数归属不清。
- 安全风险、数据迁移或发布策略尚未裁决。
- 前后端、AI、QA 或 DevOps 正在私下改变接口契约、数据模型或发布边界。

## 整合流程

1. 主会话收集所有子智能体结果。
2. 对照接口契约和架构边界检查冲突。
3. 去除重复实现，优先沉淀通用函数、组件或 service。
4. 决定合并顺序：契约和类型先合并，后端基础能力次之，前端接入随后，测试和文档最后校准。
5. 运行质量门禁。
6. 更新项目状态看板。

## 联调流程

1. 子智能体发现跨角色问题，向主会话提交联调请求。
2. 主会话登记到项目状态看板，标记提出方、处理方、影响范围和阻塞程度。
3. 主会话把请求转成处理方的 `/goal` 任务。
4. 处理方返回结论、改动、验证结果和是否影响契约。
5. 主会话裁决最终方案，并同步给相关子智能体。
6. QA 或对应验证智能体验证问题关闭。

## 并行任务状态

```text
待拆分 -> 已分配 -> 进行中 -> 联调中 -> 待整合 -> 待验证 -> 完成
```

任何子任务失败，不代表整个项目失败；主会话应判断是重试、改派、降级、串行处理还是阻塞。
