# Specialist Deliverables

本规范定义每个专业子智能体必须产出的文档和颗粒度。目标是在开发前把字段、接口格式、页面状态、错误码、权限、验收标准对齐，避免后期联调返工。

## 前期颗粒度对齐门禁

进入并行开发前，主会话必须确认以下内容已经明确或标记为待确认：

- 用户角色、核心路径、MVP 范围和非目标。
- 页面、组件、状态、交互和空/加载/错误/权限状态。
- API 路径、方法、请求字段、响应字段、错误码和权限要求。
- 数据模型、字段类型、索引、唯一约束、迁移和兼容策略。
- 前后端类型定义、mock 数据、接口契约来源。
- AI prompt、上下文输入、输出格式、审计字段和降级策略。
- 测试用例、验收标准、验证命令和缺陷分级。
- 部署环境、配置项、密钥、监控、告警和回滚方式。

未对齐的内容必须进入状态看板的“待确认接口 / 待确认字段 / 风险与阻塞”，不能假装已确定。

| --- | --- | --- |

| UI / UX | UX 流程、页面状态、设计 token、组件清单、可访问性要求 | `docs/product/UX_SPEC.md` 或 `docs/design/UI_SPEC.md` |
| Frontend | 页面路由、组件结构、状态模型、接口依赖、mock 数据、前端验证点 | `docs/engineering/FRONTEND_SPEC.md` |
| Backend | API 契约、数据模型、权限、错误码、事务、缓存、迁移 | `docs/engineering/API_SPEC.md` / `DATA_MODEL.md` |
| AI Engineer | Prompt 版本、上下文字段、模型参数、输出 schema、审计和降级 | `docs/engineering/AI_SPEC.md` |
| QA / Testing | 测试策略、测试用例、验收矩阵、自动化命令、缺陷分级 | `docs/process/QA_STRATEGY.md` / `TEST_PLAN.md` |

| Docs | README、使用说明、变更日志、发布说明 | `README.md` / `CHANGELOG.md` |

## 对齐检查

主会话在开发前必须输出对齐结论：

```text
READY / NEEDS_ALIGNMENT / BLOCKED
```

READY 条件：

- 每个专业智能体的关键文档已存在或明确不适用。
- API 字段、数据模型、错误码、权限和页面状态已对齐。
- 所有待确认项都有负责人和截止点。
- 前端、后端、QA 使用同一份契约或类型来源。
- 用户能查看每个角色的文档并理解当前项目边界。
